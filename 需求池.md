# 需求池

这个文件用于把用户语言转成可执行的产品与技术判断。原则是：不无谓增加业务量，但保证网站设计能满足真实业务需求。

## 工作原则

- 先理解业务目的，再决定技术实现。
- 每个需求都保留用户原话，避免过度翻译导致失真。
- 优先做能直接改善转化、效率、信任感或交付质量的事情。
- 不为了“功能完整”堆功能；复杂度必须服务于业务结果。
- 设计判断要克制、清晰、有取舍，像产品负责人一样对体验负责。

## 发布与测试规则

- 每次实现需求后，必须先在 dev 环境测试。
- 每次改完代码后，必须同步更新对应测试流程：说明要测哪个页面、哪个入口、哪些操作、预期结果是什么。
- 如果我无法完整执行测试，必须明确提示你需要测试哪些关键路径。
- 只有你确认 “OK” 之后，才可以提交到 GitHub，并更新到 testing service。
- 没有你的 OK，不主动 push、不创建 PR、不触发 testing service 更新。

## 执行计划

### Build registration form 优化

1. 梳理现有流程
   - 确认 trial email、save application、membership record 写入分别在哪些前后端模块。
   - 确认现有数据字段是否已经包含 start date、end date、arrival time、membership code、age group reminder。

2. 邮件与保存逻辑
   - 修改 trial 邮件模板，加入 start date、end date、arrival time。
   - 调整 save application：保存不发送用户邮件。
   - 回归检查正式提交、批准或确认类邮件仍然正常发送。

3. 会员资料写入逻辑
   - 以 membership code 作为判断依据。
   - 已存在：只更新现有 membership record 的 birthday、ageGroup 和 birthday cert file，并更新已有 club invite application。
   - 不存在：创建新 membership record。
   - 其他会员字段保持不变，避免误覆盖已有资料。
   - 避免误删申请历史，会员主档和 application 记录要分清。

4. Age group reminder 视觉强化
   - 把 reminder 放在用户做年龄组相关选择的上下文里。
   - 用更清晰的标题、边框/底色/强调色提升可见性，但避免红色 error 语义。
   - 不增加多余解释，只突出用户必须注意的规则。

5. Date of Birth 输入约束
   - 高亮 Date of Birth 下方的 age group 计算提示。
   - Date of Birth 控件限制可选日期，避免用户选择 Outside Trial Age Groups。
   - disabled input/select 要有明确锁定样式，特别是 club invite lookup 后自动填充的字段。

6. Email composer 富文本化
   - 先确认后端邮件发送是否支持 HTML body，同时保留纯文本 fallback。
   - 前端 admin email composer 从 textarea 改为轻量富文本编辑器。
   - 支持基础格式，不做复杂邮件设计器。
   - 确认 email log 能保存和展示富文本内容，避免丢格式。

## 当前判断

- R-006 是体验优化，范围应控制在视觉强调和位置优化，不需要扩成新功能。
